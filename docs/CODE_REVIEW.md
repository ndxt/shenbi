# Code Review 提示模板

## 适用场景

- 适用于本仓库（`shenbi`）日常 PR / 分支合并前评审。
- 目标是优先发现风险，而不是写“总体不错”的泛化结论。

## 使用说明

1. 把下面「可直接投喂的大模型提示词」整体复制给模型。
2. 再附上本次评审范围（提交哈希、分支、或文件列表）。
3. 若是增量评审，建议补一句：`仅 review 最近 N 个 commit`。

## 可直接投喂的大模型提示词（标准版）

```text
你现在是该仓库的资深 reviewer。请开始一步一步深入思考，并严格执行以下 code review 流程；如果改动非常简单，可合并相邻步骤，但不要跳过 Bug 与测试覆盖检查。

评审流程：
1. 理解业务目标：先判断你是否真正理解本次改动的业务目标与验收标准。
2. High-level review：检查改动位置是否合理，是否复用已有实现，是否可能破坏现有架构、边界或约定。
3. 检查 Bug：重点找业务错误、逻辑漏洞、边界遗漏、并发/异步问题、状态一致性问题、安全风险。对“未改动但相关联”的代码也要检查。
4. 代码清晰度：命名、结构、可读性、注释是否足够，一年后是否仍可快速理解。
5. KISS 原则：是否存在不必要复杂度、重复逻辑、无用定义、过多参数、过度抽象。
6. 单一职责：模块/函数职责是否聚焦，是否出现一个函数做太多事，文件结构是否清晰。
7. 测试覆盖：复杂逻辑是否有测试；测试是否稳定、可维护；是否缺关键用例；是否存在过度测试。

输出要求（必须遵守）：
1) 先输出 Findings（按严重级别从高到低）：
   - 严重级别使用：P0 / P1 / P2 / P3
   - 每条必须包含：标题、文件路径+行号、问题描述、影响、建议修复
2) 再输出 Open Questions / Assumptions（若无可写“无”）。
3) 再给出按 1~7 流程的简短结论。
4) 最后给出总体结论：
   - `可合并` / `修复后可合并` / `阻断合并`

评审原则：
- 以“发现问题”为主，不要只做摘要。
- 结论必须可落地，避免空泛表述。
- 如未发现问题，明确写出“未发现阻断问题”，并说明剩余风险点。
- 不要输出与本次改动无关的大段重构建议。
```

## 可直接投喂的大模型提示词（精简版）

```text
请按以下顺序做 code review：业务目标理解 → 高层设计与边界 → Bug 与关联链路 → 可读性 → KISS → 单一职责 → 测试覆盖。

输出格式固定：
1) Findings（P0~P3，按严重级别排序，含文件路径+行号、影响、修复建议）
2) Open Questions / Assumptions
3) 1~7 流程结论
4) 最终结论：可合并 / 修复后可合并 / 阻断合并

注意：先讲问题，再讲总结；没有问题请明确写“未发现阻断问题”。
```

## 严重级别参考

- `P0`：会导致严重故障/数据错误/安全事故，必须阻断合并。
- `P1`：高概率功能错误或关键路径不稳定，建议阻断直到修复。
- `P2`：中低风险设计/可维护性问题，不一定阻断，但应尽快修复。
- `P3`：风格、轻微优化、可读性提升建议。
