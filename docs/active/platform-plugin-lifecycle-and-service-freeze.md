# 插件平台生命周期与服务面冻结

> 更新时间：2026-03-06
> 目的：作为 Phase 3 平台基线的最终冻结结论，明确生命周期、稳定服务面、候选服务面的准入条件。

## 1. 生命周期冻结结论

### 1.1 宿主内部阶段

1. `register`
2. `activate`
3. `deactivate`
4. `dispose`

### 1.2 对插件作者暴露的最小模型

1. 插件只声明 `manifest`。
2. 插件可选实现 `activate(context)`。
3. 插件通过 `activate` 返回清理函数，宿主在 `deactivate / dispose` 阶段统一消费。
4. `register` 不是插件钩子，而是宿主内部的加载和校验步骤。

### 1.3 当前触发规则

1. `register`：宿主收集 manifest、校验 contribution、合并注册表时触发。
2. `activate`：宿主挂载插件运行时时触发。
3. `deactivate`：仅在插件被手动停用或宿主主动停用时触发。
4. `dispose`：插件卸载或宿主销毁时触发。

### 1.4 明确不做的触发场景

1. 面板隐藏不触发 `deactivate`。
2. tab 切换不触发 `deactivate`。
3. 扩展点暂时不可见不触发 `deactivate`。

## 2. 稳定服务面冻结结论

### 2.1 当前稳定服务面

1. `document`
2. `selection`
3. `commands`
4. `notifications`

### 2.2 冻结规则

1. 上述四类服务面为当前平台稳定基线。
2. 后续业务增强优先复用这四类服务面和命令总线。
3. 未通过准入评审的新能力，不进入 `PluginContext`。

## 3. 候选服务面决策

### 3.1 保留候选但暂不接入

1. `workspace`
2. `navigation`
3. `storage`
4. `clipboard`
5. `overlay`

### 3.2 暂不接入原因

1. 当前平台已能通过命令、菜单、上下文菜单、命令面板覆盖主要交互闭环。
2. 这些服务面都还缺少足够稳定的跨插件调用样本。
3. 现在直接接入 `PluginContext`，会过早固化尚未验证过的抽象。

### 3.3 下一次准入门槛

1. 至少两个插件对同一能力有真实复用需求。
2. 能明确回答“为什么不能只用 command”。
3. 能给出最小接口草案，而不是宿主私有实现直出。
4. 能补齐最小集成测试和一份迁移说明。

## 4. 命令与服务边界冻结

### 4.1 继续走命令总线的能力

1. 用户触发型动作
2. 可由快捷键、菜单、命令面板复用的动作
3. 不需要同步返回值的动作

### 4.2 才允许进入服务面的能力

1. 高频同步读写
2. 需要明确返回值
3. 需要订阅或状态查询
4. 具有明确跨插件复用价值

## 5. 本轮冻结后的平台基线

1. 命令注册表已统一宿主命令和插件命令。
2. 快捷键、菜单、上下文菜单、命令面板已复用同一条命令执行链路。
3. 命令支持 `when / enabledWhen / aliases / keywords / category / description`。
4. 菜单支持 `target / group`，上下文菜单支持 `group`。
5. 新插件已可通过模板按统一协议接入。

## 6. 下一阶段只剩可选增强

1. 搜索排序继续优化
2. 最近使用持久化
3. checked/toggled 状态
4. 更多扩展点
5. 候选服务面逐项准入
