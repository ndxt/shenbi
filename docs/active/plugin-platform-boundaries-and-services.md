# 插件平台边界与服务面设计

> 对应 Phase 3.1
> 目的：固定 Host / Platform / Plugin 三层边界，并明确哪些能力进入标准服务面，哪些继续走命令总线或留在宿主私有实现。

---

## 1. 设计目标

1. 给新增平台能力一个明确归位规则，避免能力散落在 `editor-ui`、`preview` 和各业务插件中。
2. 在 `PluginContext` 继续收口后，控制新服务面的增长速度，只把真正跨插件复用的能力纳入平台。
3. 让后续 Files / Setter / AI 增强都能在稳定协议上展开，而不是边做功能边改基础接口。

---

## 2. 三层边界

### 2.1 Host Layer

位置：

1. `packages/editor-ui`
2. `apps/preview` 中的最终装配与宿主接线

职责：

1. 负责壳层布局、面板容器、键盘事件监听、主题与窗口级交互。
2. 负责把宿主能力适配为标准服务面，注入 `PluginContext`。
3. 负责插件生命周期调度、命令分发、扩展点渲染。

不负责：

1. 不承载 Files / Setter / AI 的具体业务逻辑。
2. 不向单个插件暴露私有宿主对象。

### 2.2 Platform Layer

位置：

1. `packages/editor-plugins/api`

职责：

1. 定义 `PluginContext`、contribution 类型、生命周期约定和快捷键协议。
2. 提供跨插件复用的标准服务面。
3. 提供兼容 helper、贡献收集与协议层工具函数。

不负责：

1. 不直接渲染 UI。
2. 不持有具体业务状态。

### 2.3 Plugin Layer

位置：

1. `packages/editor-plugins/files`
2. `packages/editor-plugins/setter`
3. `packages/editor-plugins/ai-chat`
4. 后续新增插件包

职责：

1. 消费标准服务面，声明 contributions，实现具体业务能力。
2. 通过命令、快捷键、面板等协议接入宿主。

不负责：

1. 不直接监听全局键盘事件。
2. 不直接依赖宿主私有组件或内部状态结构。

---

## 3. 归位规则

新增能力按下面顺序判断：

1. 如果它是窗口级 UI、全局事件、宿主渲染容器能力，归 Host。
2. 如果它会被两个及以上插件复用，且需要稳定协议，归 Platform。
3. 如果它只服务某一个插件域，归 Plugin。

补充规则：

1. 仅因为“现在写在宿主里最方便”不能成为进入 Host 的理由。
2. 仅因为“未来可能有用”不能成为进入 Platform 的理由。
3. 新能力若无法回答“第二个插件是谁”，原则上先不要进入标准服务面。

---

## 4. 当前稳定服务面

### 4.1 `document`

职责：

1. Schema 读取
2. Schema 替换
3. 节点 patch
4. Schema 订阅

为什么进入平台：

1. Files、Setter、AI 都依赖它。
2. 高频同步读写，不适合只靠命令总线。

结论：

1. 保留为核心标准服务面。
2. 后续扩展优先走子服务，不新增顶层 alias。

### 4.2 `selection`

职责：

1. 获取当前选中节点
2. 获取当前选中节点 ID
3. 选区变化订阅

为什么进入平台：

1. Setter、AI、后续上下文菜单都会依赖选区状态。
2. 需要同步读和订阅能力。

结论：

1. 保留为核心标准服务面。

### 4.3 `commands`

职责：

1. 统一命令执行入口

为什么进入平台：

1. 快捷键、菜单、命令面板最终都要落到命令执行。
2. 命令总线天然是跨插件触发能力的公共协议。

结论：

1. 保留为核心标准服务面。
2. 后续所有“可触发动作型”扩展点都应围绕 `commands` 建立。

### 4.4 `notifications`

职责：

1. 统一消息通知出口

为什么进入平台：

1. 多插件需要一致的用户反馈入口。
2. 不应让插件直接依赖具体 UI 库。

结论：

1. 保留为标准服务面。
2. 初版维持 fire-and-forget 语义。

## 5. 候选服务面评估

### 5.1 `workspace`

候选职责：

1. 当前文件 ID
2. 脏状态
3. 工作区模式
4. 最近打开文件

判断：

1. Files、Command Palette、后续导航类能力都会依赖。
2. 这些是平台级状态，不应散落在插件内部自行维护。

结论：

1. 进入下一批标准服务面。
2. 初版先做只读查询 + 必要订阅。
3. 若后续差异过大，再拆分 `workspace.files` 和 `workspace.mode`。

### 5.2 `navigation`

候选职责：

1. 打开指定侧栏
2. 激活指定面板
3. 切换辅助面板
4. 打开命令面板

判断：

1. 这是宿主 UI 编排能力，但会被多个插件复用。
2. 若不平台化，插件会反向依赖宿主实现细节。

结论：

1. 进入下一批标准服务面。
2. 只暴露声明式导航意图，不暴露宿主组件实例。

### 5.3 `storage`

候选职责：

1. 插件偏好保存
2. 本地缓存
3. 简单 KV 读写

判断：

1. 多插件会反复需要。
2. 若没有统一入口，容易出现各插件各写一套 localStorage 包装。

结论：

1. 进入下一批标准服务面。
2. 初版只做异步 KV 能力，不承担复杂同步逻辑。

### 5.4 `clipboard`

候选职责：

1. 节点复制
2. Schema 片段复制
3. 编辑器内部粘贴

判断：

1. 这是编辑器语义，不等同于浏览器文本剪贴板。
2. 若直接暴露浏览器 Clipboard API，会把“文本复制”和“节点复制”混在一起。

结论：

1. 进入下一批标准服务面。
2. 初版只做编辑器内部结构化复制粘贴。
3. 系统剪贴板需求后续单独评估。

### 5.5 `overlay`

候选职责：

1. confirm
2. prompt
3. modal / drawer 打开意图

判断：

1. 这是明显的跨插件交互需求。
2. 插件不应直接持有 UI 库 API。

结论：

1. 进入下一批标准服务面。
2. 只暴露意图接口和 Promise 结果，不暴露底层组件实现。

## 6. 哪些能力不进入标准服务面

以下能力当前不进入 `PluginContext`：

1. 只服务某个插件的局部表单状态
2. 特定面板内部 UI 组件引用
3. 单个插件专有的数据转换工具
4. 宿主私有布局实现细节
5. 未来可能会用到、但当前没有第二个消费者的能力

原则：

1. 先留在插件内。
2. 只有出现明确复用和边界稳定需求时，再提升为平台能力。

---

## 7. 服务 vs 命令判定规则

### 7.1 进入服务面的能力

满足以下两个条件中的大多数：

1. 高频
2. 需要同步返回值
3. 需要订阅能力
4. 多插件复用
5. 与某类平台状态强关联

典型例子：

1. `document.getSchema`
2. `selection.getSelectedNodeId`
3. `workspace.getCurrentFile`

### 7.2 继续走命令总线的能力

满足以下两个条件中的大多数：

1. 低频触发
2. 更像动作而不是状态
3. 可由快捷键、菜单、命令面板统一触发
4. 不要求同步返回值

典型例子：

1. `file.saveSchema`
2. `editor.undo`
3. `commandPalette.open`

### 7.3 判定结论

1. 读状态、订阅状态、同步 patch 倾向走服务面。
2. 执行动作、触发流程、跨扩展点调用倾向走命令总线。
3. 同一能力不同时提供“私有宿主调用”和“公开平台服务”两套入口。
