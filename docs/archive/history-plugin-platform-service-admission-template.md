## 1. 使用方式

当准备新增一个平台服务面时，先按下面模板填写，再决定是否进入 `PluginContext`。

若以下问题无法回答清楚，默认不进入平台，先留在插件内或继续走命令总线。

## 3. 评审门槛

满足以下条件中的大多数，才建议进入标准服务面：

1. 有两个及以上明确消费者。
2. 存在同步读取或订阅需求。
3. 仅靠命令总线会显著增加复杂度。
4. 宿主适配简单，不会引入大规模胶水代码。
5. 能清楚定义“负责什么”和“不负责什么”。

出现以下任一情况，默认不进入：

1. 只有一个消费者。
2. 只是为了“写起来方便”。
3. 只是未来可能有用，但没有现实复用方。
4. 会直接暴露宿主私有对象或具体 UI 库 API。

## 4. 示例：`workspace`

### 4.1 基本信息

- 服务面名称：`workspace`
- 归属阶段：Phase 3
- 预期消费者：Files、Command Palette、后续 Workspace Surface

### 4.2 能力定义

- 解决问题：统一提供当前文件、脏状态、最近文件等平台级上下文
- 最小接口：
  - `getCurrentFile()`
  - `isDirty()`
  - `subscribe(listener)`

### 4.3 为什么不能只用命令

- Files 和命令面板都可能高频读取当前文件状态。
- 若只走命令查询，会产生重复调用和状态分散。
- 需要订阅能力时，命令模型不合适。

### 4.4 结论

- 建议进入标准服务面。
- 初版只做只读查询与订阅，不承载打开/保存等动作。

## 5. 示例：`overlay`

### 5.1 基本信息

- 服务面名称：`overlay`
- 归属阶段：Phase 3
- 预期消费者：Files、AI、后续任何需要确认框/提示框的插件

### 5.2 能力定义

- 解决问题：统一提供 confirm / prompt / modal 调起入口
- 最小接口：
  - `confirm(options): Promise<boolean>`
  - `prompt(options): Promise<string | undefined>`

### 5.3 为什么不能只用命令

- 它本质上需要返回用户决策结果。
- 单纯命令执行不适合建模“打开 UI 并等待返回值”的交互。

### 5.4 结论

- 建议进入标准服务面。
- 只暴露声明式意图接口，不暴露宿主具体组件实例。
